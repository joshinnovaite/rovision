# ST-02.5: Upload, Seed-On-Startup, And Cache Headers

Parent: [[BOW-02-Backend]]
Status: done
Progress: done — POST /api/upload streams sha256 -> hit(200)/miss(202); startup seeds bundles; detections served with immutable Cache-Control.

## Preconditions
- [[ST-02.2-Ingest-And-Rollup]] and [[ST-02.3-Read-Endpoints]] done (ingest + read paths work).
- Hero bundles available on disk to seed from.

## Steps
1. Add `POST /api/upload`: **stream** the uploaded file, computing `sha256` incrementally (no full
   buffering of the large clip).
2. On **hit** (hash exists in `videos`): return `200` with the matched video row/hash.
3. On **miss** (no row): return `202` gracefully — do **not** create a row (cache-only model; there
   is no live inference to run).
4. Add seed-on-startup: at app start, ingest each hero bundle whose `hash` is absent from `videos`
   (reuse `ingest.ingest_bundle`).
5. Add gzip to `GET /detections` and immutable/long-lived cache headers to `/detections` and `/clip`.
6. Confirm the hash uses the **same bytes** the seed hashed, so uploading the handed `raw.mp4` hits.

## Files
- `app/backend/routes.py` — `upload` route (stream-hash).
- `app/backend/main.py` — startup hook calling seed-on-startup.
- Response/middleware config for gzip + cache headers.

## Acceptance
- On startup, hero bundles are ingested if absent (visible in `GET /videos`).
- Re-uploading the same `raw.mp4` returns `status:"hit"` (200); an unknown file returns `202` with no new row.
- `GET /detections` is gzipped and cacheable.
