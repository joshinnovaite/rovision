# ST-02.4: Range-Seekable Clip Serving

Parent: [[BOW-02-Backend]]
Status: done
Progress: done — Starlette FileResponse returns 206 with content-range to a Range request; full GET 200.

## Preconditions
- [[ST-02.2-Ingest-And-Rollup]] done: `data/static/<hash>/raw.mp4` and the overlay mp4 exist.
- The clips were encoded with `+faststart` in [[ST-01.2-Raw-Clip-Encode-And-Fixture-Capture]]
  (moov atom at front — required for seeking to work).

## Steps
1. Add `GET /api/videos/{hash}/clip?variant=raw|overlay`.
2. Resolve the file: `variant=raw` → `static/<hash>/raw.mp4`; `variant=overlay` →
   `static/<hash>/internal_assets_demo.mp4` (404 on unknown hash/variant).
3. Serve via **Starlette `FileResponse`** (not a manual streamer) so HTTP `Range` / `206 Partial
   Content` is handled correctly — mandatory for browser `<video>` seeking.
4. Set `media_type='video/mp4'` and an `Accept-Ranges: bytes` capable response.
5. Keep this endpoint **isolated** from the JSON routes (it is the risky bit).

## Files
- `app/backend/routes.py` — `clip` route using `from starlette.responses import FileResponse`.

## Acceptance
- `curl -H 'Range: bytes=0-1' .../clip?variant=raw` returns **HTTP 206** with a `Content-Range`.
- The clip **seeks** in a real browser `<video>` (faststart + 206 together — the key risk).
